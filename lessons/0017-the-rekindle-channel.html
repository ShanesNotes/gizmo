Gizmo · Lesson 0017 · Path A refactor

The rekindle channel — a state machine, proven in isolation

0016 deleted the old win. Now we build the shape of the new one: a Beacon that goes Dormant → Rekindling → Rekindled as Gizmo holds its radius. We prove the whole machine headless, by hand-ticking positions — and on purpose it still doesn't win the run. That wire is 0018.

Why build the machine before the win

The risky, novel thing in Path A isn't "set phase = COMPLETE" — we did that in one line for years. It's the area-hold: a thing that fills while you're somewhere, pauses when you leave, and can't be cheesed by tapping it once. So we de-risk that first, in pure isolation, where a test can hand-feed Gizmo's position and check every edge. Wiring it to the win (0018) is then trivial and safe, because the hard part is already pinned.

Part 1 — the design we grilled

Before any code, we settled the one behaviour that defines the channel: what happens when Gizmo leaves the radius mid-channel. Two options — PAUSE (the number freezes) or DECAY (it drains while you're away). Decay is the more interesting game (the "cold world pushes back"), but it's just a number rule on top of a working pause — not a different machine. So:

LEAVE-RADIUS    PAUSE now; DECAY is the agreed later upgrade (same machine, one rule swapped)
STATE vs NUMBER leaving touches ONLY the number; Rekindling is a one-way door (never back to Dormant)
FILL            linear, constant rate
DURATION        BEACON_CHANNEL_SECONDS = 8.0  — a placeholder, retuned vs the real siege (0018/0023)

Choosing "leaving touches only the number" is what makes decay-later free: decay becomes progress -= rate*dt while outside, and the state machine proved here never changes.

Part 2 — the machine (≈12 lines inside Simulation)

The whole channel is one method, called from tick() after the enemies/weapon/pickups. It lives inside Simulation (ADR 0002 — the sim owns rules), not the controller:

func _update_beacon(dt, gizmo_position):
    if beacon_radius <= 0.0 or beacon_state == BEACON_REKINDLED:
        return                          # inert until authored; Rekindled is terminal
    var to_beacon = gizmo_position - beacon_position;  to_beacon.y = 0.0
    if to_beacon.length() > beacon_radius:
        return                          # OUTSIDE → PAUSE: hold the bank, keep Rekindling (one-way door)
    if beacon_state == BEACON_DORMANT:
        beacon_state = BEACON_REKINDLING      # the door opens, once
    beacon_channel_progress = min(1.0, beacon_channel_progress + dt / BEACON_CHANNEL_SECONDS)
    if beacon_channel_progress >= 1.0:
        beacon_state = BEACON_REKINDLED       # NOTE: the WIN is wired in 0018, not here

Read the two early returns as the two design decisions made physical: the first keeps the channel inert when no beacon is placed; the second is PAUSE (leave → return with the bank untouched, and never a state reset).

Inert by default — why the balance harness didn't move

beacon_radius defaults to 0.0, so a sim with no authored beacon skips the channel entirely — exactly like obstacles being empty by default. That's not a nicety: it's why the 300 s decent-kite balance run (open floor, no beacon) stayed at PASS 38. A new system that silently shifted the balance numbers would be a refactor smell; this one provably doesn't touch them.

Part 3 — proving it by hand-ticking

Because Simulation is pure RefCounted, a test is the player: it feeds a position each tick. "Inside" is Vector3.ZERO (the beacon centre); "outside" is far away. Six tests pin the machine, including the ones that catch the cheese:

entry never instant   one tick inside → Rekindling, 0 < progress < 1   (can't tap-to-win)
fills while held      8.5 s inside    → progress ≥ 1.0, Rekindled
pauses + one-way      in 3 s, out 4 s → bank HELD (±0.0001), state still Rekindling
resumes after return  in → out → in   → completes (the bank survived the trip)
no win yet            Rekindled        → phase is STILL playing  (0018 wires the win)

The "no win yet" test is the honest one: it asserts the thing we deliberately didn't do, so a future edit that wires the win early can't slip past unnoticed.

Done means…

Done = the Beacon state machine exists inside Simulation, fills as an area-hold, pauses on leave without ever reverting, can't be entered instantly, and does not end the run. Verify: godot --headless --path godot --script res://tests/run_simulation_tests.gd → PASS 86 (+15), full gate 151, balance unchanged at 38. You can explain why we proved the machine before the win, why "leaving touches only the number" makes decay a free later upgrade, and why inert-by-default kept the balance harness honest.
