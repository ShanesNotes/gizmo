Your First Headless Test

Turn yesterday's temporary print() into a real test — one command that checks the game state and tells you passed or failed. This is the loop every later lesson rides on.

0004 ✓ → 0005 · first headless test → 0006 · the tick

The one idea

A test is just a script that builds your game state, checks each value against what it should be, and reports. Green means the rules are correct; a red FAIL line tells you exactly which value drifted. Because the simulation is pure data (Lesson 0004), we can check it with no window and no scene — Godot runs the script straight from the terminal.

From here on, "does it work?" stops being "squint at the screen" and becomes one command that's green or red. That's what lets us port fast without breaking what already works.

This is the locked workflow. The seed's mechanics are the source of truth, and headless tests make each ported rule provable in isolation (decision ADR-002). It's the one place we leave the editor on purpose — pure logic deserves a fast, windowless check.

Do it

  1. In the FileSystem dock, right-click the tests/ folder → Create New → Script…
  2. Set Inherits to SceneTree (type it in — a SceneTree script is one Godot can run on its own from the command line). Path: res://tests/run_simulation_tests.gd → Create.
  3. Replace the file with this and save (Ctrl+S): 
    extends SceneTree

const SimulationScript = preload("res://scripts/simulation.gd")

var _failures := 0

func _init() -> void:
    var sim = SimulationScript.new()
    var state: Dictionary = sim.create_game_state()

    _assert_equal(state["phase"], "playing", "run starts in the playing phase")
    _assert_equal(state["world"]["width"], 2600, "world width matches the seed")
    _assert_equal(state["world"]["height"], 1700, "world height matches the seed")
    _assert_approx(state["run_duration"], 240.0, "a run lasts 240 seconds")
    _assert_approx(state["player"]["x"], 1300.0, "player starts centered on x")
    _assert_approx(state["player"]["y"], 850.0, "player starts centered on y")
    _assert_equal(state["player"]["level"], 1, "player starts at level 1")
    _assert_equal(state["player"]["xp"], 0, "player starts with no xp")
    _assert_equal(state["enemies"].size(), 0, "no enemies at start")
    _assert_equal(state["pickups"].size(), 0, "no pickups at start")

    if _failures == 0:
        print("Simulation tests passed")
    else:
        printerr("Simulation tests failed: %d" % _failures)
    quit(_failures)

func _assert_equal(actual, expected, label: String) -> void:
    if actual != expected:
        _failures += 1
        printerr("FAIL %s: expected %s, got %s" % [label, str(expected), str(actual)])

func _assert_approx(actual, expected: float, label: String, epsilon := 0.0001) -> void:
    if abs(float(actual) - expected) > epsilon:
        _failures += 1
        printerr("FAIL %s: expected %.4f, got %.4f" % [label, expected, float(actual)])
  4. Run it from a terminal at the repo root: 
    godot --headless --path godot --script res://tests/run_simulation_tests.gd
    You should see Simulation tests passed and an exit code of 0.

What each piece does

extends SceneTreeA script Godot can launch directly with --script — no main scene, no window. func _init()The entry point. It runs once when the SceneTree script starts. preload(...)Loads your simulation.gd so the test can build state from it. _assert_equal / _assert_approxCompare actual vs expected; on mismatch, bump _failures and print a FAIL line. approx is for floats (never compare floats with ==). quit(_failures)Exit code = number of failures. 0 = pass, so terminals and CI know the result.

Prove it actually bites

A test that can't fail is worthless. Make it go red on purpose: change one expectation — say 240.0 to 999.0 — and rerun. You'll get: 

FAIL a run lasts 240 seconds: expected 999.0000, got 240.0000

Now change it back and confirm green again. That red→green flip is the whole game of testing.

That's the win. One command now proves Gizmo's opening state is exactly the seed's — automatically, in a fraction of a second. Every rule you port from here gets a line in this file, and you'll always know the instant something breaks.
Tip: that run command is long. You don't have to memorize it — it's saved in CLAUDE.md under "Commands and gates," and your AI teacher can run it for you any time. Still removing the temp print from title.gd? Do that now; tests are our checks from here on.

Mental model to keep

  • A test = build state → assert values → report. Green or a precise red.
  • Pure logic tests headless — no scene, no window, near-instant.
  • Exit code 0 = pass. That's how tools know without reading the text.
  • Floats use approx, not == — rounding makes exact equality unreliable.
  • Always see a test fail once before trusting its pass.
