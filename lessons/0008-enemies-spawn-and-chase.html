Gizmo · Lesson 0008 · The loop

Enemies: the first pressure machine

Spawn one archetype, make it chase Gizmo, and let contact hurt — the first thing in the game that pushes back.

Why this, now

You can move, the run can be won or lost — but nothing threatens Gizmo yet. Enemies are the run's first pressure machine: they force kiting decisions, they're what will drop Sparks and trigger the level-up dopamine later, and they're the foundation for waves → elites → bosses. For the horde loop we're after (Vampire Survivors / Diablo / Megabonk), enemies must be balance data first — spawn cadence, chase speed, contact damage — not one-off scene scripts. Tuning intent: reference/game-balance-reference.md §5 (scaling/horde) and §3 (defense); mechanics: simulation.ts (ENEMY_SPECS, updateEnemies).

The architecture decision (ADR 0002)

This is where the tested Simulation finally meets the running game. We chose a hybrid: Simulation owns the rules (enemies live there as data — spawn, chase, contact, HP), and the scene renders them. A new GameController node bridges the two: each physics frame it feeds Gizmo's position into Simulation.tick(dt, gizmo_position), then mirrors each enemy's data onto a visual node. Gizmo's own movement stays in Godot (the feel you built in 0004–0005). And we use simple seek steering, not NavigationAgent3D — open-floor horde pressure with many cheap agents is clearer and testable; navigation can come later when maps have obstacles. Full rationale: docs/adr/0002-simulation-owns-rules-scene-renders.md.

How this maps to the balance reference
  • Spawn cadence (spawn_interval) → §5.2 (time is the scaling axis — "the clock is the boss") + §5.3 (horde tuning). For now a flat interval; real ramping pressure is 0010 (waves).
  • Contact damage → reuses 0007's take_damage (§3.4), so i-frames and the lose condition already apply for free.
  • Enemy stats (the nibbler) → ENEMY_SPECS (simulation.ts:259). We keep the relative balance (damage 1) but express positions/speeds in Godot metres, not Phaser pixels (ADR 0002).

Part 1 — enemies as data agents (logic, test-first)

An enemy is just a little bag of numbers. In godot/scripts/simulation.gd, an inner class:

class Enemy extends RefCounted:
	var position := Vector3.ZERO
	var hp: float = 1.0
	var speed: float = 0.0
	var radius: float = 1.0
	var damage: int = 1
	var kind: String = ""

Then tick grows a second argument — where Gizmo is — and runs the enemies. (The default keeps the 0006/0007 tests, which call tick(dt), working.)

func tick(dt: float, gizmo_position := Vector3.ZERO) -> void:
	if phase != PHASE_PLAYING:
		return
	var safe_dt := clampf(dt, 0.0, MAX_DT)
	elapsed += safe_dt
	invulnerable = maxf(0.0, invulnerable - safe_dt)
	_update_enemies(safe_dt, gizmo_position)
	if elapsed >= run_duration:
		phase = PHASE_COMPLETE

func _update_enemies(dt: float, gizmo_position: Vector3) -> void:
	_spawn_timer += dt
	if _spawn_timer >= spawn_interval:
		_spawn_timer -= spawn_interval
		_spawn_nibbler(gizmo_position)
	for e in enemies:
		var to_player := gizmo_position - e.position
		to_player.y = 0.0  # stay on the floor plane
		var dist := to_player.length()
		if dist > 0.0001:
			e.position += to_player / dist * e.speed * dt  # seek toward Gizmo
		if dist <= e.radius:
			take_damage(e.damage)  # contact reuses the 0007 primitive (i-frames!)

_spawn_nibbler drops one enemy on a ring around Gizmo, at a deterministic angle (golden-angle spread, so tests don't depend on RNG). The seek is the whole "AI": a unit vector toward Gizmo times speed times dt. Contact just calls take_damage — which means i-frames and the lose condition from 0007 apply automatically.

Test-first — spawn, then chase, then contact

The tests grow in the same order the behavior does (add them to the runner, call from _initialize()):

func _test_enemy_spawns_on_cadence() -> void:
	var sim := Sim.new()
	sim.spawn_interval = 0.2
	_check_eq("no enemies at the start", sim.enemies.size(), 0)
	for i in 4:
		sim.tick(0.05, Vector3.ZERO)  # 0.20s -> one spawn
	_check_eq("one enemy after one interval", sim.enemies.size(), 1)
	for i in 4:
		sim.tick(0.05, Vector3.ZERO)
	_check_eq("two enemies after two intervals", sim.enemies.size(), 2)

func _test_enemy_chases_gizmo() -> void:
	var sim := Sim.new()
	sim.spawn_interval = 0.05
	var target := Vector3(5.0, 0.0, 0.0)
	sim.tick(0.05, target)
	var e := sim.enemies[0]
	var before := e.position.distance_to(target)
	sim.tick(0.05, target)
	_check("enemy moves toward Gizmo", e.position.distance_to(target) < before)

func _test_enemy_contact_damages_player() -> void:
	var sim := Sim.new()
	sim.spawn_interval = 0.05
	sim.tick(0.05, Vector3.ZERO)
	var e := sim.enemies[0]
	var hp_before := sim.hp
	sim.tick(0.05, e.position)  # put Gizmo on the enemy -> contact
	_check_eq("contact costs the player 1 hp", sim.hp, hp_before - 1)
	sim.tick(0.05, e.position)  # again immediately -> blocked by i-frames
	_check_eq("contact respects i-frames", sim.hp, hp_before - 1)

func _test_no_spawning_after_run_over() -> void:
	var sim := Sim.new()
	sim.spawn_interval = 0.05
	sim.take_damage(100)  # -> gameover
	for i in 10:
		sim.tick(0.05, Vector3.ZERO)
	_check_eq("a dead player spawns no enemies", sim.enemies.size(), 0)

Run the suite (godot --headless --path godot --script res://tests/run_simulation_tests.gd). Implement spawn → the cadence checks go green; add the seek → "moves toward Gizmo" goes green; route contact through take_damage → the damage/i-frame checks go green. End state: PASS — 39 checks. (Unlike 0007, these checks build on each other — the chase test needs a spawned enemy to chase — so they go green spawn-first, not all-at-once.)

Part 2 — render the enemies (the scene)

The logic is the truth; the scene just draws it. Two pieces:

  1. A visual — godot/scenes/enemy.tscn: a Node3D root (the agent's ground position) with a small reddish SphereMesh child lifted to sit on the floor. No physics, no script — it's pure visual; the controller moves it.
  2. The bridge — godot/scripts/game_controller.gd on a GameController node in godot/scenes/main.tscn:
extends Node3D

const EnemyScene := preload("res://scenes/enemy.tscn")

@export var gizmo: Node3D   # assign Gizmo in the Inspector

var sim := Simulation.new()
var _views: Array[Node3D] = []  # one visual per sim enemy, parallel by index

func _physics_process(delta: float) -> void:
	var gizmo_position := gizmo.global_position if gizmo != null else Vector3.ZERO
	sim.tick(delta, gizmo_position)

	while _views.size() < sim.enemies.size():   # a visual for each new enemy
		var view: Node3D = EnemyScene.instantiate()
		add_child(view)
		_views.append(view)

	for i in sim.enemies.size():                 # mirror positions
		_views[i].global_position = sim.enemies[i].position

In godot/scenes/main.tscn, add a GameController node and set its Gizmo export to the Gizmo node. Press Play: red nibblers spawn on the ring and converge on Gizmo. Stand still and they'll swarm and kill you (run → gameover); move and you can kite them — the first real decision the game asks of you.

Deliberately simple, for now

Enemies don't die yet (no weapons — that's 0009 combat), don't avoid each other (they stack on a stationary target — separation is later polish), and spawn on a flat cadence (ramping pressure is 0010 waves). Spark drops on kill come with combat. The index-parallel _views works because enemies only grow in 0008; once they can die, the controller switches to a per-enemy map with removal.

Done means…

Done = the suite is green (39 checks), and in-game red enemies spawn and chase Gizmo, hurting him on contact (HP falls; standing still ends in gameover). You can explain the ADR 0002 split — Simulation owns the enemy rules, the GameController renders them — and why we used seek steering, not navigation. Then I log 0008, and 0009 gives Gizmo a way to fight back (enemies take damage, die, and drop Sparks — closing the loop back to 0006's leveling).
