Gizmo · Lesson 0016 · Path A refactor

Strip the timer-win — delete the old loop before building the new one

The first move of the Path A arc (0016→0023): turn survive-the-timer into traverse-and-rekindle-a-Beacon. Today we don't add the Beacon — we remove the old win, let the gate go red on it, and let the test suite prove we deleted exactly what we meant to. The run ends today intentionally un-winnable.

Why delete first — code gravity

ADR 0005 names the real risk in its own words: "a finish-line timer and a destination are different games, and code gravity pulls back toward the arena unless the win condition itself moves." If we built the Beacon while the old elapsed >= run_duration win still sat there, every run could still be won by standing still — the arena would quietly survive. So the disciplined first step is to make the old win honestly disappear. The test suite is what keeps that honest.

Part 1 — the old win lived in three places

A win condition is never one line. It's a rule plus every test that pins the rule. Before deleting, we found all three:

RULE   simulation.gd:204         if elapsed >= run_duration: phase = PHASE_COMPLETE
PIN A  run_simulation_tests.gd   _test_run_completes        — clock elapses → PHASE_COMPLETE
PIN B  run_balance_tests.gd      _test_decent_kite_can_complete_the_run — a fair run reaches PHASE_COMPLETE

Miss a pin and the gate stays green for the wrong reason. Find them all and the gate becomes a tripwire: delete the rule, and exactly those two pins must go red — no more, no fewer.

Part 2 — red first (the deletion is load-bearing)

We delete the two-line rule and run the gate before fixing anything. The red is the evidence:

SIM      FAIL - run completes (a win) when the timer elapses (got playing, expected complete)
         FAIL - tick is a no-op once the run is over
BALANCE  FAIL - decent true-speed kite can win naturally
         FAIL — 69 passed, 2 failed   ·   FAIL — 37 passed, 1 failed

Exactly the two pins, nothing else. That clean red is the proof the win was load-bearing and we located all of it. (If an unexpected test had broken, that would mean a hidden dependency on the timer-win we hadn't accounted for.)

Part 3 — rewrite the intent, don't flip a flag

Green is not "make the red go away." It's "assert the new truth." Both pins get rewritten to say what Path A means:

PIN A  _test_run_completes  ->  _test_timer_no_longer_wins
       crossing run_duration now keeps the run PHASE_PLAYING.
       Its old "tick is a no-op once the run is over" tail was meaningless
       (the run never ends now) — rewritten, not kept.

PIN B  _test_decent_kite_can_complete_the_run  ->  _test_decent_kite_survives_the_clock
       no win (no Beacon yet) — but a competent kite must still SURVIVE the
       full pressure clock. The real win returns in lesson 0018.

This is the difference between a refactor and a hack: the test now describes the game we're building, so it will keep guarding the new loop instead of rubber-stamping the deletion.

The honest catch — don't quietly weaken a guard

Removing the win meant the kite no longer stops at 240 s; it now runs the harness to its 300 s cap — 60 s longer under peak pressure. My first rewrite silently dropped the old "takes some but not lethal damage (2–6 events)" fairness band. That's a real regression guard — dropping it to go green is exactly the kind of quiet erosion to catch. So we measured the longer run instead of guessing: damage_events = 4, hp = 3, elapsed = 300. The 2–6 band still holds, so it's restored at full strength — the guard survives the refactor, it isn't loosened to fit it.

Done means…

Done = the timer-win rule is deleted; the gate is green again on the rewritten truth; the run is intentionally un-winnable (HP-0 loss still works, no Beacon yet); and no fairness guard was weakened to get there. Verify: godot --headless --path godot --script res://tests/run_simulation_tests.gd & …/run_balance_tests.gd → 71 + 38, full gate 136. You can explain why we deleted the win before building the new one, why a clean red proves the deletion was complete, and why a rewritten test guards the new loop while a flipped flag wouldn't.
