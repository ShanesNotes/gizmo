The Player Moves

Wire custom movement actions into the simulation and watch a display-only PlayerAvatar3D follow the tested state through SimSpace — speed-capped, bounded, camera following. Gizmo becomes playable without duplicating movement rules.

0007 ✓ → 0008 · the player moves → 0009 · pickups & XP

The one idea

Input flows into the simulation, never around it. Each physics frame the scene reads movement actions into a direction, hands it to update_state as input, and the sim moves the player — then SimSpace maps flat x/y onto the stage and PlayerAvatar3D displays the player's new position. The rules (speed, bounds, normalized diagonals) live in the tested sim; the scene only translates keys in and pixels out.

Ported from the seed. This is simulation.ts's movement core (lines ~603–663): normalize the input direction, ease velocity toward a target speed, integrate by dt, and clamp to the world margins. We port the faithful movement core now; dash/boost stays deferred.

Do it — part 1: movement in the simulation

In scripts/simulation.gd, add the movement constants by the others:

add with the other const lines
const PLAYER_WORLD_MARGIN: float = 70.0
const BASE_MOVE_SPEED: float = 266.0
const SPRINT_SPEED_GAIN: float = 24.0
const EVOLVED_SPRINT_GAIN: float = 34.0

Give the player velocity + facing in create_game_state() — add these inside the "player" dictionary:

inside the "player": { … } block
"vx": 0.0,
"vy": 0.0,
"facing_x": 1.0,

Then add the movement block to update_state, right after the state["elapsed"] = … line:

inside update_state, after advancing elapsed
_update_player(state, input, safe_dt)

Then add the helper below update_state(). Keeping it separate makes the movement rule easy to test and keeps scene code out of the simulation.

func _update_player(state: Dictionary, input: Dictionary, dt: float) -> void:
    var player: Dictionary = state["player"]
    var input_vector := Vector2(float(input.get("x", 0.0)), float(input.get("y", 0.0)))
    var magnitude: float = input_vector.length()
    var direction: Vector2 = input_vector / magnitude if magnitude > 0.0 else Vector2.ZERO
    var upgrades: Dictionary = state.get("upgrades", {})
    var evolved: Dictionary = state.get("evolved", {})
    var sprint_rank: int = int(upgrades.get("sprint", 0))
    var sprint_evolved: bool = bool(evolved.get("sprint", false))
    var speed: float = BASE_MOVE_SPEED + float(sprint_rank) * SPRINT_SPEED_GAIN + (EVOLVED_SPRINT_GAIN if sprint_evolved else 0.0)
    var target_velocity: Vector2 = direction * speed
    var response: float = 23.0 if magnitude > 0.0 else 30.0
    var blend: float = 1.0 - exp(-response * dt)
    var velocity := Vector2(float(player.get("vx", 0.0)), float(player.get("vy", 0.0))).lerp(target_velocity, blend)
    var next_position := Vector2(float(player["x"]), float(player["y"])) + velocity * dt
    next_position.x = clampf(next_position.x, PLAYER_WORLD_MARGIN, WORLD_WIDTH - PLAYER_WORLD_MARGIN)
    next_position.y = clampf(next_position.y, PLAYER_WORLD_MARGIN, WORLD_HEIGHT - PLAYER_WORLD_MARGIN)
    if next_position.x <= PLAYER_WORLD_MARGIN or next_position.x >= WORLD_WIDTH - PLAYER_WORLD_MARGIN:
        velocity.x = 0.0
    if next_position.y <= PLAYER_WORLD_MARGIN or next_position.y >= WORLD_HEIGHT - PLAYER_WORLD_MARGIN:
        velocity.y = 0.0
    if absf(velocity.x) > 8.0:
        player["facing_x"] = 1.0 if velocity.x > 0.0 else -1.0
    player["vx"] = velocity.x
    player["vy"] = velocity.y
    player["x"] = next_position.x
    player["y"] = next_position.y
    state["player"] = player

What the math does

input_vector.length()Length of the input vector. Dividing by it normalizes — so diagonals are not faster. velocity.lerp(target_velocity, blend)Ease toward the target speed instead of snapping instantly. clampf(next_position.x, …)Integrate position by the safe time-step, then clamp inside the world margins. facing_x = …Remember which way we're heading (for flipping the sprite later).

Do it — part 2: prove it headless

Before wiring keys, lock the rules in. Add to tests/run_simulation_tests.gd, before the pass/fail report:

add inside _init()
var s2 = sim.create_game_state()
sim.update_state(s2, {"x": 1.0, "y": 0.0}, 0.05)
var expected_blend: float = 1.0 - exp(-23.0 * 0.05)
var expected_vx: float = 266.0 * expected_blend
_assert_approx(s2["player"]["vx"], expected_vx, "right input eases toward move speed")
_assert_approx(s2["player"]["x"], 1300.0 + expected_vx * 0.05, "moving right advances x")
_assert_equal(s2["player"]["facing_x"], 1.0, "moving right faces right")

var s3 = sim.create_game_state()
sim.update_state(s3, {"x": 1.0, "y": 1.0}, 0.05)
var diagonal_velocity := Vector2(float(s3["player"]["vx"]), float(s3["player"]["vy"]))
_assert_approx(diagonal_velocity.length(), 266.0 * expected_blend, "diagonal input keeps normalized speed")

var s4 = sim.create_game_state()
s4["player"]["x"] = 2529.0
sim.update_state(s4, {"x": 1.0, "y": 0.0}, 0.05)
_assert_approx(s4["player"]["x"], 2530.0, "player clamps to the right wall")
godot --headless --path godot --script res://tests/run_simulation_tests.gd

Simulation tests passed — movement is correct before a single pixel moves.

Do it — part 3: see it, drive it

Now the scene. The important split is: Simulation owns movement; SimSpace maps coordinates; PlayerAvatar3D displays it; Main translates input. Target tree:

Main — Node3D (main.gd)
 ├─ World — Node3D
 │  ├─ Ground — MeshInstance3D
 │  └─ Entities
 │    └─ PlayerAvatar — PlayerAvatar3D instance
 ├─ CameraRig
 │  └─ Camera3D — orthographic
 └─ HUD — CanvasLayer
    └─ Root → Title — Label
  1. Add movement actions: Project → Project Settings → Input Map. Add move_left, move_right, move_up, move_down. Bind them to A/D/W/S or arrow keys.
  2. Add scripts/sim_space.gd: this is the only place that knows how flat simulation x/y becomes 3D stage x/z.
  3. Create scenes/player.tscn: root PlayerAvatar as Node3D; add a child Visuals with a simple MeshInstance3D placeholder under it. Attach scripts/player_avatar_3d.gd.
  4. Add two tiny presenters: attach scripts/camera_rig_3d.gd to CameraRig and scripts/hud_presenter.gd to HUD. They keep camera/HUD display work out of Main.
  5. Use PlayerAvatar3D.apply_snapshot(player): 
    class_name PlayerAvatar3D
extends Node3D

@onready var _visuals: Node3D = $Visuals

func apply_snapshot(player: Dictionary) -> void:
    global_position = SimSpace.to_world_from_snapshot(player)
    var facing_x: float = float(player.get("facing_x", 1.0))
    _visuals.rotation.y = 0.0 if facing_x >= 0.0 else PI
  6. Update main.gd so it reads custom input actions, ticks the sim in _physics_process, then applies the mapped snapshot: 
    extends Node3D

var sim: Simulation = Simulation.new()
var state: Dictionary = {}

@onready var player_avatar: PlayerAvatar3D = %PlayerAvatar
@onready var camera_rig: CameraRig3D = $CameraRig
@onready var hud: HudPresenter = $HUD

func _ready() -> void:
    state = sim.create_game_state()
    _apply_state()

func _physics_process(delta: float) -> void:
    sim.update_state(state, _read_input_state(), delta)
    _apply_state()

func _read_input_state() -> Dictionary:
    var direction: Vector2 = Input.get_vector(&"move_left", &"move_right", &"move_up", &"move_down")
    return {"x": direction.x, "y": direction.y}

func _apply_state() -> void:
    var player: Dictionary = state["player"]
    var player_stage_position: Vector3 = SimSpace.to_world_from_snapshot(player)
    player_avatar.apply_snapshot(player)
    camera_rig.follow_stage_position(player_stage_position)
    hud.apply_state(state)
  7. Press Play (F5) and drive with your movement keys. The simulation moves the player; SimSpace maps it; the avatar follows; the orthographic camera follows the raw SimSpace target; the HUD stays pinned. You are playing Gizmo.

Scene-side pieces

Input.get_vector(...)Reads your custom movement actions into one direction vector — already length-capped at 1. SimSpace.to_world...Maps sim x/y to stage x/z. This boundary keeps Simulation flat. PlayerAvatar3D.apply_snapshot()A display adapter: it shows simulation state without owning movement rules. CameraRig3DA display-only camera presenter follows the raw SimSpace target, not cosmetic avatar motion. HudPresenterA screen-space presenter updates the HUD without adding rules to Main.
That's the win. Keys → simulation → SimSpace → 2.5D pixels, with the rules safely test-backed. Gizmo is a real, playable, orthographic stage now — everything from here (enemies, pickups, dashes) rides this same input-to-state-to-screen path.
Bot looks plain? The mesh/avatar is just a placeholder until we import the real sparkbot sprite. Keep it simple for now — the lesson is input → simulation → 2.5D display.

Mental model to keep

  • Input goes into the sim. The scene reads keys → input; the sim decides the new flat position.
  • Normalize direction so diagonal isn't faster; multiply by speed; integrate by dt; clamp to bounds.
  • SimSpace maps x/y to x/z. Godot y is visual height only; it does not enter simulation state.
  • PlayerAvatar3D displays state; CameraRig3D follows it; HudPresenter updates screen-space UI.
  • Test the rule, then show it. Movement was green before it was visible.
