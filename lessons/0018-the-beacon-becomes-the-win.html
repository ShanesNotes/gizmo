Gizmo · Lesson 0018 · Path A refactor

The Beacon becomes the win — and a driver that can actually reach it

0017 built the rekindle channel but left it un-winnable on purpose. Now we wire the last inch: a full channel sets PHASE_COMPLETE. The catch — our balance "kite" runs away from danger and would never approach a beacon, so we also author a seek-and-hold driver to prove a fair run can truly rekindle.

Why this is two wins, not one

"Set phase = COMPLETE when the channel fills" is one line. But a green unit test only proves the mechanism fires when you hand-feed a perfect hold. It says nothing about whether a player can get there. Our existing balance proof — the "decent kite" — is built on pure evasion: every tick it steers away from the nearest enemies. A driver whose whole strategy is "flee" will never walk into a beacon and stand there. So the win needs two proofs: the sim-side wire (deterministic), and a driver that seeks and holds (reachability).

Part 1 — the sim-side wire (red-first)

0017's living test asserted "Rekindled but not a win." 0018 flips that exact assertion — the test file is the one living source of truth, so we rewrite it to the new reality and watch it go red, then green:

// the win wire, inside _update_beacon (Simulation owns the rule — ADR 0002)
if beacon_channel_progress >= 1.0:
    beacon_state = BEACON_REKINDLED
    phase = PHASE_COMPLETE         // 0018: Beacon Rekindled IS the win (ADR 0005)

Two tests pin it: a full channel reaches PHASE_COMPLETE, and — the one that protects the player — death with a beacon present still loses.

Loss wins the race — the phase guard

_update_beacon runs after enemies in the same tick. So on the tick where Gizmo's last HP drains and the channel would complete, two outcomes collide. Loss must win (ADR 0005: "loss is unchanged"). One line settles it — the method bails if the run already ended this tick:

if beacon_radius <= 0.0 or beacon_state == BEACON_REKINDLED or phase != PHASE_PLAYING:
    return        // a death earlier this tick already set GAMEOVER — don't overwrite it

The test sits a heavy mob on the beacon: the channel is filling, but contact damage kills first → GAMEOVER, not a sneaky win.

Part 2 — a driver that seeks and holds

To prove the win is reachable, we author a new balance profile that does what the evasion kite can't: walk to the Beacon and stay in its radius while the auto-fire clears whatever wanders in.

const SEEK_BEACON := Vector3(0, 0, 5);  const SEEK_BEACON_RADIUS := 3.0
// each tick: step toward the beacon centre at GIZMO_SPEED; once there (<0.4 m), hold.
var to_beacon = SEEK_BEACON - pos;  to_beacon.y = 0
if to_beacon.length() > 0.4:  pos += to_beacon.normalized() * GIZMO_SPEED * DT
sim.tick(DT, pos)

Result: it reaches the radius in ~0.6 s, holds 8 s, and rekindles at elapsed 8.6 s with HP to spare. The test pins all four facts — PHASE_COMPLETE, state Rekindled, channel actually full, and HP ≥ 1 — plus that the finish lands on the channel timer (8–20 s), never instantly.

Honest scope — this is reachability, not the climax

The rekindle lands at 8.6 s, when pressure is near zero — so this test proves the win is achievable by fair play, not that it's a hard finale. Making the rekindle a real siege (the swarm at peak pressure is the boss) is lesson 0023, where entering Rekindling overrides spatial exposure to peak. We say that out loud rather than dressing an easy hold up as a climax.
