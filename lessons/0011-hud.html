Gizmo · Lesson 0011 · The HUD

HUD: show the state

Five lessons of hidden state finally surface — a brass nameplate with cyan HP, a violet level gem, Sparks, the run clock, and a level-up flash off the 0009 event seam.

Why this, now

0006–0010 built a lot of state — HP, XP, level, the run clock, the pressure curve — but it all lived inside the Simulation, invisible. 0011 puts it on screen, themed to the canonical target design-handoff/gizmo-hud.png (the gouache cosmos: brass frames, cyan energy, the violet spark gem). It also cashes in the last_events seam from 0009 — the level-up flash. The architecture holds (ADR 0002): the Simulation is the truth, the scene renders it; the HUD just reads.

v1-scoped — only what has data

The full mockup shows a lot (Spark-of-Humanity meter, Core Matrix abilities, Gadgets, Scrap, minimap, objective) — but those drive systems we haven't built. Building their chrome now would be dead UI. So 0011 themes only the data-backed pieces: the Gizmo nameplate + HP bar, the level gem, XP/Sparks, and a run-clock readout where the old "WAVE x/5" sat (ART_DIRECTION + ADR 0003 — no wave counter). The rest land with their systems.

Part 1 — the HUD scene (brass, themed)

The HUD is a CanvasLayer (it floats above the 3D world) holding a Control tree, anchored into the screen corners so it survives any resolution. The brass look is a Theme resource (scenes/hud_theme.tres) of StyleBoxFlats — beveled borders, riveted corners, dark translucent fills — with the two energy fills (cyan HP, violet XP) set per-bar:

Hud (CanvasLayer)               scripts/hud.gd
└─ Root (Control, Full Rect)
   ├─ Nameplate   (PanelContainer, Top-Left)   → "GIZMO" · HpBar (cyan) · HpLabel "7 / 7"
   ├─ LevelCluster(VBox,          Top-Center)  → GemBadge (violet) "1" · XpBar (violet)
   ├─ SparksReadout(PanelContainer,Top-Left ↓) → gem · SparksLabel "0" · "SPARKS"
   ├─ TimerPanel  (PanelContainer, Top-Right)  → TimerLabel "4:00"
   └─ LevelUpFlash(Label,          Center)     → "LEVEL UP" (hidden until it fires)

Each widget the script touches is marked unique (%HpBar, %LevelLabel…) so hud.gd can reach it by name without brittle paths. The scene ships with sensible default values, so running hud.tscn by itself shows a populated HUD — handy for tuning the look against the mockup.

Part 2 — drive it from the controller

The HUD never reaches into the Simulation. Instead the controller pushes one frame of state into it each tick — one call, after the world is synced:

# game_controller.gd
@export var hud: Hud   # assign a hud.tscn instance in the Inspector (optional)

func _physics_process(delta: float) -> void:
	var gizmo_position := gizmo.global_position if gizmo != null else Vector3.ZERO
	sim.tick(delta, gizmo_position)
	_sync(sim.enemies, _enemy_views, EnemyScene)
	_sync(sim.pickups, _spark_views, SparkScene)
	if hud != null:
		hud.render(sim)   # one-way: state in, pixels out

And render(sim) is a flat mapping — read a value, set a widget. No game logic lives in the HUD:

# hud.gd
func render(sim: Simulation) -> void:              # typed: the controller→HUD contract
	_hp_bar.value = sim.hp_progress() * 100.0
	_hp_label.text = "%d / %d" % [sim.hp, sim.max_hp]
	_level_label.text = str(sim.level)
	_update_xp_fill(sim.xp_progress() * 100.0)    # eases the fill — see below
	_sparks_label.text = str(sim.xp)              # Sparks ARE the xp currency (ADR 0001)
	_timer_label.text = format_clock(sim.time_remaining())
	for event in sim.last_events:                 # the 0009 seam, finally consumed
		if event.get("type") == "levelup":
			_play_level_up_flash()
			break

Part 3 — the run clock (test-first)

The one piece of pure logic in the HUD is formatting the clock, so it gets a test (its own runner, run_hud_tests.gd; the visual layout is verified by eye). It's a count-down, so it rounds up — a run with 0.4s left still reads 0:01, never a premature 0:00:

static func format_clock(seconds: float) -> String:
	# count-DOWN clock: round up, so any time left still shows (0.1s -> "0:01")
	var total: int = 0 if seconds <= 0.0 else int(ceilf(seconds))
	return "%d:%02d" % [total / 60, total % 60]
func _test_format_clock() -> void:
	_check_eq("zero -> 0:00", Hud.format_clock(0.0), "0:00")
	_check_eq("5s -> 0:05 (zero-padded)", Hud.format_clock(5.0), "0:05")
	_check_eq("65s -> 1:05", Hud.format_clock(65.0), "1:05")
	_check_eq("125s -> 2:05", Hud.format_clock(125.0), "2:05")
	_check_eq("full run 240s -> 4:00", Hud.format_clock(240.0), "4:00")
	_check_eq("negative clamps to 0:00", Hud.format_clock(-5.0), "0:00")
	_check_eq("0.1s left rounds up -> 0:01", Hud.format_clock(0.1), "0:01")   # count-down
	_check_eq("59.1s left rounds up -> 1:00", Hud.format_clock(59.1), "1:00")

godot --headless --path godot --script res://tests/run_hud_tests.gd → PASS — 8 checks.

The level-up flash — the seam pays off

0009 had the Simulation record a per-tick last_events list (attack/hit/defeat/pickup/ levelup) that nothing read yet. Now it does: when render sees a levelup event, it shows the centered "LEVEL UP" label and tweens its alpha 1 → 0 over 1.2s (killing any prior tween, so back-to-back level-ups restart cleanly). No state-diffing, no guessing — the HUD just reacts to what the Simulation reported.

The XP bar fills — it doesn't snap

Sparks should feel earned, so the XP bar eases toward each new fill instead of jumping. render never pokes the bar's value directly — it calls _update_xp_fill(target), which snaps on the very first frame (no startup animation) and after that retargets a short TRANS_CUBIC / EASE_OUT tween whenever the fill changes. It's a no-op when the value is unchanged, so it's safe to call every frame. (The HP bar still snaps — damage should read instantly; only the reward bar gets the flourish.)

Integration update

Originally this HUD was a one-step editor wiring task. After 0012's playtest follow-up, GameController now auto-instances scenes/hud.tscn if its HUD slot is empty, so the playable scene shows state even while main.tscn is changing in the art stream. Manual wiring still works later and overrides the default.

Deferred — the rest of the mockup

Named, not built (each waits for its system): the Spark of Humanity meter (its mechanic is still TBD — ADR 0001), the Core Matrix ability bar, Gadgets, the Scrap currency, the minimap, and the objective card. The brass theme is a StyleBoxFlat family, not literal filigree texture — the art pipeline can swap richer frames into the same Theme without touching hud.gd.

Done means…

Done = hud.tscn renders the brass HUD; the controller's one render(sim) call maps state→widgets (HP / level / XP / Sparks / clock); the XP bar eases as it fills; a level-up flashes; and format_clock is green (8 checks). That's verified standalone and via a controller probe — the HUD lights up in the running game the moment you finish the one wiring step above.
