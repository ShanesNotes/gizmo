Gizmo · Lesson 0004 · The character

Move Gizmo with code

Give the clanker a body and a brain — and drive him around the floor with keyboard or controller. This is the one where it becomes a game.

Why this, now

Right now Gizmo is a statue: a beautiful model that can't do anything. This lesson is the turning point — your first GDScript, your first Input Map, and your first physics body, all aimed at one payoff: press a key, the clanker moves. Everything after this (enemies, combat, the whole loop) is built on this exact pattern. It's the biggest lesson so far; we'll take it in four small phases.

Three ideas before we build

1. A character needs a body, not just a mesh

The model you instanced is pure visuals. To move-and-collide, Gizmo needs to live inside a CharacterBody3D — a node built for exactly this: you set its velocity, then call move_and_slide() and the engine moves it and handles bumping into things. So we'll wrap the model in a CharacterBody3D and make Gizmo his own reusable scene — the same shape every enemy will use later. (Heads-up: the floor is still only a visual mesh, not a collider — for this slice Gizmo just glides along the ground plane at y=0. The capsule we add isn't catching him on the floor yet; it's groundwork for colliding with enemies and walls in later lessons.)

2. Input is named, not hard-coded

We never write "if the W key is down." We define an action called move_up in the project's Input Map and bind W (and ↑) to it. The code asks "is move_up active?" That's what lets players rebind keys later, and lets a gamepad stick feed the same action.

3. Movement is world-relative (because the camera is fixed)

Here's the payoff of 0002. Because the camera never rotates, "up on screen" is always the same world direction. So we map input straight onto the ground: left/right → world X, up/down → world Z. No camera math. (A first-person game would have to rotate input by where you're looking — we're spared that entirely.)

Phase A — give Gizmo a body (new scene)
  1. Scene → New Scene. Choose Other Node → CharacterBody3D. Rename the root Gizmo.
  2. Ctrl+S → save as scenes/gizmo.tscn.
  3. Drag assets/gizmo.glb onto the Gizmo root to add the model as a child (this is the visual).
  4. Select Gizmo, add a child CollisionShape3D. In the Inspector, set its Shape → New CapsuleShape3D; set the capsule's Radius ≈ 0.4, Height ≈ 2.0. Set the CollisionShape3D's Position Y to 1.0 so the capsule wraps his body.
Phase B — the input actions (already set up: keyboard + Xbox)

Binding an analog stick by hand is fiddly and easy to get subtly wrong, so the four actions are already defined in the project — each bound to both keyboard and an Xbox controller. Open Project → Project Settings → Input Map to see them:

  • move_up → W, ↑, left-stick up, D-pad up
  • move_down → S, ↓, left-stick down, D-pad down
  • move_left → A, ←, left-stick left, D-pad left
  • move_right → D, →, left-stick right, D-pad right

Two things worth understanding while you look at them:

  • One action, many inputs. Each action holds several events, and the script never cares which one fired — key, stick, or D-pad all activate move_up identically. That is the entire reason we use named actions instead of raw keys.
  • Deadzone (0.2). A real stick never rests at perfect zero, so the deadzone ignores tiny drift — Gizmo won't creep when you're not touching it. The stick is analog, so a gentle push moves him slowly; get_vector passes that magnitude straight through.

Poke around to learn the panel: add a binding with +, or change a deadzone. The keys are bound by physical position, so WASD stays put on non-QWERTY layouts. (Want to bind these yourself from scratch instead? Say so and I'll walk you through the empty version.)

Phase C — the movement script
  1. Select the Gizmo root (the CharacterBody3D) → click the Attach Script icon → path res://scripts/gizmo.gd → Create. (Godot makes the scripts/ folder for you as it saves the file.)
  2. Replace the template with this, then save:
extends CharacterBody3D

## How fast Gizmo moves, in metres per second.
@export var speed: float = 6.0
## How quickly he reaches top speed and how hard he brakes.
@export var acceleration: float = 40.0
@export var friction: float = 50.0

func _physics_process(delta: float) -> void:
	# 1. Read input as a 2D vector: x = left/right, y = up/down on screen.
	var input := Input.get_vector("move_left", "move_right", "move_up", "move_down")
	# 2. Map it onto the ground plane: X = sideways, Z = toward/away from camera.
	var direction := Vector3(input.x, 0.0, input.y)
	# 3. Accelerate toward target velocity, or brake to a stop.
	if direction != Vector3.ZERO:
		velocity.x = move_toward(velocity.x, direction.x * speed, acceleration * delta)
		velocity.z = move_toward(velocity.z, direction.z * speed, acceleration * delta)
	else:
		velocity.x = move_toward(velocity.x, 0.0, friction * delta)
		velocity.z = move_toward(velocity.z, 0.0, friction * delta)
	# 4. Let the engine move him and resolve collisions.
	move_and_slide()

Reading it: _physics_process runs at a fixed physics rate (the right place for movement — _process would jitter). get_vector hands back a tidy direction (already normalized on diagonals, so you don't move faster going NE). move_toward eases velocity up to speed and back down to zero, giving weight instead of an on/off snap. We touch x and z but never y — he stays on the ground. move_and_slide() does the actual moving.

Phase D — put him in the game
  1. Open scenes/main.tscn. Delete the old Gizmo (the raw .glb instance from 0003).
  2. Drag scenes/gizmo.tscn onto Main to instance your new character scene. Set its Position to (0, 0, 0).
  3. Press F5 and drive with WASD or your controller's left stick. 🎮

If something's off
  • He doesn't move: action names must match the script exactly (move_up, not Move Up). Check the Input Map spelling.
  • Up/down feel reversed: that's a preference — swap the W/S bindings, or tell me and we'll flip the Z sign.
  • A yellow warning triangle on the CharacterBody3D before you added the collision shape is normal; it clears once the CapsuleShape3D is in.
  • He slides forever / too floaty: raise friction; for snappier starts raise acceleration. They're exported, so you can tweak them right in the Inspector while the game's stopped.

How you'll know it worked

WASD (or the left stick) glides Gizmo around the floor — diagonals included — and he eases to a stop when you let go, all seen down the fixed Diablo camera. He won't turn to face his direction yet (he'll moonwalk sideways); that's 0005, along with the camera learning to follow him as he leaves the middle of the floor.

Done means…

Done = you can drive Gizmo with the keyboard, and you can explain why we map input to world X/Z directly instead of using his facing direction. Then I log 0004 and we make him turn toward travel and the camera follow him (0005).
