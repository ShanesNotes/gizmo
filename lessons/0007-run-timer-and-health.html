The run timer & health

Give the run a clock you can win and an HP bar you can lose — still headless, still test-first.

Why this, now

0006 gave Gizmo progression; now the run needs stakes: a way to win (survive the bounded run) and a way to lose (run out of health). Both already live in simulation.ts — the run clock and the player's HP — so we keep porting, headless and test-first, growing the same Simulation. Same truth hierarchy as before: tuning intent in reference/game-balance-reference.md (this slice draws on §3 Defense/Sustain and §5/§12 enemy-scaling — mapped below), meaning in design-handoff/NARRATIVE.md, mechanics in simulation.ts.

HP is not the Spark of Humanity (ADR 0001)

This lesson ports the player's HP — the health bar (hp/max_hp); taking damage lowers it, zero ends the run. That is not the Spark of Humanity meter (a separate objective/survival meter Gizmo keeps alive — "Keep it safe. Keep it alive.") whose mechanics we haven't designed yet. Three distinct things — HP, Sparks, Spark of Humanity — per docs/adr/0001-sparks-hp-spark-of-humanity-are-distinct.md. We build HP here; the Spark of Humanity is deliberately deferred.

How this maps to the balance reference
  • RUN_DURATION / elapsed / run_progress → §5.2 (time as the enemy-scaling axis — "the clock is the boss") + §12.1 (choose the genre axis first).
  • hp / max_hp → §3.1 (HP = the raw defensive pool, best vs one-shots).
  • HIT_INVULN (i-frames) → §3.4 (one-shot vs sustain / chip-damage control).
  • future hearts & recovery → §3.3 (recovery cap policy) — deferred.

The slice: a clock and a health bar

Two pure mechanics from simulation.ts:

  • The run clock. Time advances each frame; the run is bounded at RUN_DURATION = 240s (simulation.ts:202). When it elapses, you win (phase = "complete", simulation.ts:489-493). run_progress and time_remaining mirror lines 533 and 535. We clamp the frame time to [0, 0.05] — the source uses min(dt, 0.05) (simulation.ts:463) so a lag spike can't fast-forward the run, and flooring at 0 also stops a stray negative frame from rewinding it.
  • Health. A real hit lowers HP, grants brief i-frames (1.58s, so one touch isn't instant death), and floors at 0; reaching 0 is a lose (phase = "gameover", simulation.ts:722-738).

A small phase field ties it together: playing → either complete (won) or gameover (lost). The clock only ticks while playing (simulation.ts:460).

240s is a seed value, not the genre target

We port RUN_DURATION = 240 because that is the Phaser seed — faithful to the source. But it's a v1 micro-run, not the final horde-survival length: the balance reference's horde preset puts run length at 20–30 min (§13.1). The real length gets tuned later alongside spawn pressure, XP pace, and wave structure (§12.1 build order). A short clock just keeps the test loop fast for now.

What this slice does NOT include

This is the base HP/gameover primitive only. The enemy-contact geometry, the "no damage for the first 7s" grace, knockback, and the secondWind one-time save the source runs before gameover (simulation.ts:732-736) all arrive in later enemy/recovery lessons. So "faithful" here means faithful to the primitive, not the whole contact block. (Enemies in 0008 are what will actually call take_damage.)

func _test_run_clock() -> void:
	var sim := Sim.new()
	sim.run_duration = 1.0
	for i in 5:
		sim.tick(0.05)  # 5 * 0.05 = 0.25s elapsed
	_check("run_progress ~ 0.25", absf(sim.run_progress() - 0.25) < 0.001)
	_check("time_remaining ~ 0.75", absf(sim.time_remaining() - 0.75) < 0.001)

func _test_dt_is_clamped() -> void:
	var sim := Sim.new()
	sim.tick(10.0)  # a huge frame must not skip the whole run
	_check("dt clamped to 0.05", absf(sim.elapsed - 0.05) < 0.0001)

func _test_run_completes() -> void:
	var sim := Sim.new()
	sim.run_duration = 0.08
	sim.tick(0.05)
	_check_eq("still playing mid-run", sim.phase, Sim.PHASE_PLAYING)
	sim.tick(0.05)  # elapsed 0.10 >= 0.08
	_check_eq("run completes (a win) when the timer elapses", sim.phase, Sim.PHASE_COMPLETE)
	var before := sim.elapsed
	sim.tick(0.05)
	_check("tick is a no-op once the run is over", sim.elapsed == before)

func _test_damage_and_iframes() -> void:
	var sim := Sim.new()  # hp 7, invulnerable 0
	var landed := sim.take_damage(2)
	_check("first hit lands", landed)
	_check_eq("hp drops by the damage", sim.hp, 5)
	var blocked := sim.take_damage(2)
	_check("i-frames block the next hit", not blocked and sim.hp == 5)
	sim.run_duration = 999.0  # don't let the run complete while we wait out i-frames
	for i in 40:
		sim.tick(0.05)  # 2.0s > 1.58 i-frame window
	sim.take_damage(2)
	_check_eq("damage lands again after i-frames", sim.hp, 3)

func _test_damage_zero_and_negative() -> void:
	var sim := Sim.new()
	var z := sim.take_damage(0)
	_check("zero damage is a no-op (no free i-frames)", not z and sim.hp == 7 and sim.invulnerable == 0.0)
	var n := sim.take_damage(-5)
	_check("negative damage is a no-op", not n and sim.hp == 7 and sim.invulnerable == 0.0)

func _test_negative_dt_does_not_rewind() -> void:
	var sim := Sim.new()
	sim.tick(0.05)
	sim.tick(-1.0)  # a negative frame must not rewind the clock
	_check("negative dt cannot rewind the clock", absf(sim.elapsed - 0.05) < 0.0001)

func _test_death_is_gameover() -> void:
	var sim := Sim.new()
	sim.take_damage(100)
	_check_eq("hp floors at 0", sim.hp, 0)
	_check_eq("reaching 0 hp is a gameover (lose)", sim.phase, Sim.PHASE_GAMEOVER)
	var landed := sim.take_damage(1)
	_check("no damage applies once the run is over", not landed)

func _test_hp_progress() -> void:
	var sim := Sim.new()
	_check("full hp bar reads 1.0", absf(sim.hp_progress() - 1.0) < 0.001)
	sim.take_damage(2)  # 5 / 7
	_check("hp bar reads 5/7 after a hit", absf(sim.hp_progress() - 5.0 / 7.0) < 0.001)

Step 2 (GREEN) — the real run clock & health

Replace the stubs:

## Advance the run by dt seconds. Mirrors the top of updateGameState
## (simulation.ts:459-494): no-op unless playing; dt clamped; the run COMPLETES
## (a win) when the timer runs out.
func tick(dt: float) -> void:
	if phase != PHASE_PLAYING:
		return
	var safe_dt := clampf(dt, 0.0, MAX_DT)  # also guards a negative dt rewinding the run
	elapsed += safe_dt
	invulnerable = maxf(0.0, invulnerable - safe_dt)
	if elapsed >= run_duration:
		phase = PHASE_COMPLETE

func run_progress() -> float:           # simulation.ts:533
	return clampf(elapsed / run_duration, 0.0, 1.0)

func time_remaining() -> float:         # simulation.ts:535
	return maxf(0.0, run_duration - elapsed)

func hp_progress() -> float:            # HP bar fill, for the HUD (balance §3.1)
	return clampf(float(hp) / max_hp, 0.0, 1.0)

## A real hit lowers HP, grants i-frames, floors at 0; 0 HP ends the run (lose).
## Mirrors simulation.ts:722-738 (minus enemy geometry / secondWind). Zero or
## negative damage is a no-op — it must not hand out free i-frames (balance §3.4).
## HP, not the Spark of Humanity (ADR 0001). Returns true if the hit landed.
func take_damage(amount: int) -> bool:
	if amount <= 0:
		return false
	if phase != PHASE_PLAYING or invulnerable > 0.0:
		return false
	hp = maxi(0, hp - amount)
	invulnerable = HIT_INVULN
	if hp <= 0:
		phase = PHASE_GAMEOVER
	return true

Why it's shaped this way

  • A tiny phase machine. playing → complete | gameover is the whole run lifecycle. Win and lose are just two phase transitions; the HUD and screens (0011/0012) will read phase.
  • The dt clamp is a real rule, so we test it. Without bounding the frame time, a one-second hitch would jump the clock a full second (or end the run); a negative frame could even rewind it. We clamp to [0, 0.05] and pin both ends with tests.
  • i-frames make HP fair. 1.58s of invulnerability after a hit means brushing an enemy costs one tick of HP, not all of it — which is why we tick time forward in the test to prove the window clears.
  • Reject bad input at the boundary. take_damage ignores zero and negative amounts; otherwise a blocked or 0-damage hit would hand out free i-frames once armor, shields, or dodges exist (balance §3.4). Public methods guard their inputs.
  • i-frames will need to be felt, not just computed. Later they pair with visible/audible feedback — a flash, a sound, a brief knockback, a HUD pulse — so the player can read the window. Here we only build the timing.
  • Still RefCounted, still headless. A whole run can be simulated and checked with no window — the same testability that made 0006 fast.
