Gizmo · Lesson 0006 · The logic

Sparks & leveling, test-first

Leave the scene behind: port Gizmo's first real rule — collecting Sparks and re-humanizing — and prove it with a test before it exists.

Why this, now

The player feels good, but there's no game yet — no progression, stakes, or enemies. Those rules already exist, and in this project truth has a clear order:

  • Design / tuning intent → reference/game-balance-reference.md (§5 Progression & XP). The north star for how it should feel.
  • Fiction / meaning → design-handoff/NARRATIVE.md (§4 system→fiction). What each number means in Gizmo's world.
  • Mechanics implementation → game-src-phaser/src/game/simulation.ts. One concrete, battle-tested implementation of the above — the code we port from.

(Orientation lives in CONTEXT.md.) Because this logic is pure math — no models, no camera — we develop it headless and test-first: write a failing test, then the code to pass it. That's how we port ~1,700 lines of rules without guesswork.

What this is, in Gizmo's fiction

In the code it's called xp — because that's what the Phaser implementation uses. In Gizmo's world it's Sparks: fragments of the human spark that Gizmo rescues (NARRATIVE.md §4, "Sparks = xp pickups… the thing being preserved"). And leveling isn't generic RPG growth — it's Gizmo re-humanizing himself with what he saved ("Level up → upgrade draft… re-humanizing himself"). We'll keep the code name xp (faithful to the source) but think of it as Sparks.

Don't confuse three separate things (the HUD shows them as distinct elements — see design-handoff/ART_DIRECTION.md, and docs/adr/0001-sparks-hp-spark-of-humanity-are-distinct.md):

  • Sparks — the xp currency you collect (this lesson).
  • HP — the player's health bar (damage, death). A later lesson.
  • The Spark of Humanity meter — a separate objective/survival meter Gizmo keeps alive ("Keep it safe. Keep it alive."). Its mechanics aren't designed yet — it is not the same as HP, and not the same as Sparks.

The idea: red → green

Test-Driven Development is a tight loop:

  1. Red — write a test that states what the code should do; run it; it fails.
  2. Green — write the smallest code that makes it pass; run it; it passes.
  3. Refactor — clean up, with the test as your safety net.

Writing the test first forces you to decide what "correct" means before you're tangled in implementation — and leaves a permanent check that survives every future change.

Our slice: Sparks & leveling

From simulation.ts we port: the curve nextXpForLevel(level) (Sparks per level, simulation.ts:1721-1725); the level-up rule where the remainder carries (simulation.ts:1029-1038); and xpProgress (how full the Sparks bar is, simulation.ts:534) which the HUD will need. The starting state is the player's level/xp/nextXp (simulation.ts:342-362). Balance §5.1 classifies this kind of curve as the polynomial family (note the level^1.8 term).

Paths: speak from the repo root

The Godot project lives under godot/. So files are godot/scripts/simulation.gd and godot/tests/run_simulation_tests.gd. Inside the engine those same files are res://scripts/… and res://tests/… (res:// = the godot/ project root, which the run command sets with --path godot). Create them under godot/…; don't make a root-level scripts/ or tests/.

Step 1 (RED, part 1) — write the test runner first

No heavy framework needed. Create godot/tests/run_simulation_tests.gd — a tiny runner Godot executes headlessly. _check_eq prints got-vs-expected so a failure tells you exactly what's wrong:

extends SceneTree

# Run with:
#   godot --headless --path godot --script res://tests/run_simulation_tests.gd
# Exits 0 if all pass, 1 if any fail.

var _passed := 0
var _failed := 0

# Load by path so this runner works headless without an editor import step
# (a raw --script run doesn't refresh the global class_name cache).
const Sim := preload("res://scripts/simulation.gd")

func _initialize() -> void:
	print("Running simulation tests…")
	_test_next_xp_vectors()
	_test_xp_below_threshold()
	_test_level_up_carries_remainder()
	_test_xp_progress()
	_test_negative_ignored()
	print("")
	if _failed == 0:
		print("PASS — %d checks" % _passed)
		quit(0)
	else:
		printerr("FAIL — %d passed, %d failed" % [_passed, _failed])
		quit(1)

func _check(desc: String, condition: bool) -> void:
	if condition:
		_passed += 1
		print("  ok   - ", desc)
	else:
		_failed += 1
		printerr("  FAIL - ", desc)

func _check_eq(desc: String, actual, expected) -> void:
	if actual == expected:
		_passed += 1
		print("  ok   - ", desc)
	else:
		_failed += 1
		printerr("  FAIL - %s (got %s, expected %s)" % [desc, actual, expected])

# Exact vectors pin the formula faithfully (simulation.ts:1721-1725).
func _test_next_xp_vectors() -> void:
	_check_eq("next_xp lvl 1", Sim.next_xp_for_level(1), 92)
	_check_eq("next_xp lvl 2", Sim.next_xp_for_level(2), 188)
	_check_eq("next_xp lvl 3", Sim.next_xp_for_level(3), 291)
	_check_eq("next_xp lvl 5", Sim.next_xp_for_level(5), 595)
	_check_eq("next_xp lvl 10", Sim.next_xp_for_level(10), 1811)

func _test_xp_below_threshold() -> void:
	var sim := Sim.new()
	var leveled := sim.add_xp(10)
	_check("a few Sparks do not level you", not leveled)
	_check_eq("still level 1", sim.level, 1)
	_check_eq("Sparks banked", sim.xp, 10)

func _test_level_up_carries_remainder() -> void:
	var sim := Sim.new()
	var leveled := sim.add_xp(100)  # level-1 threshold is 92
	_check("crossing the threshold levels up", leveled)
	_check_eq("now level 2", sim.level, 2)
	_check_eq("remainder carries (100 - 92)", sim.xp, 8)
	_check_eq("next threshold is level 2's", sim.next_xp, Sim.next_xp_for_level(2))

func _test_xp_progress() -> void:
	var sim := Sim.new()
	sim.add_xp(46)  # ~half of 92
	_check("Sparks bar reads ~half full", absf(sim.xp_progress() - 0.5) < 0.05)

func _test_negative_ignored() -> void:
	var sim := Sim.new()
	sim.add_xp(-5)  # pickups are never negative; guard against it
	_check_eq("negative Sparks are ignored", sim.xp, 0)

Step 2 (GREEN) — fill in the real logic

Replace simulation.gd with the faithful port:

class_name Simulation
extends RefCounted

## Headless game logic, ported from game-src-phaser/src/game/simulation.ts.
## First slice: Sparks (the `xp` currency in code) and re-humanizing (leveling).

## Re-humanizing progress. "Sparks" in fiction; `xp` in the Phaser code.
var level: int = 1     # simulation.ts:359
var xp: int = 0        # Sparks banked toward the next level (simulation.ts:360)
var next_xp: int = 0   # Sparks needed for the next level (simulation.ts:361)

func _init() -> void:
	next_xp = next_xp_for_level(level)

## Sparks needed to advance FROM `lvl` to the next level.
## Faithful to nextXpForLevel() — simulation.ts:1721-1725.
static func next_xp_for_level(lvl: int) -> int:
	var base := 56.0 + lvl * 14.0 + pow(lvl, 1.8) * 12.5
	var pacing := 1.12 if lvl == 1 else 1.48 + minf(0.36, (lvl - 2) * 0.07)
	return int(floor(base * pacing))

## Bank Sparks. Returns true if it triggered a level-up (re-humanizing).
## Mirrors the level-up block — simulation.ts:1029-1038 — remainder carries over.
## Pickup values are non-negative, so we guard.
func add_xp(amount: int) -> bool:
	xp += maxi(0, amount)
	if xp >= next_xp:
		xp -= next_xp
		level += 1
		next_xp = next_xp_for_level(level)
		return true
	return false

## How full the Sparks bar is, 0..1 — for the HUD. Faithful to xpProgress() — simulation.ts:534.
func xp_progress() -> float:
	return clampf(float(xp) / next_xp, 0.0, 1.0)

Why it's shaped this way

  • extends RefCounted — no Node, no scene. Pure logic that runs and is tested with no game window. That's the headless win, and RefCounted frees itself (no manual cleanup).
  • Faithful to the source. The formula, the carry rule, and xp_progress are copied from simulation.ts (lines cited above), not reinvented. When in doubt, the TypeScript wins; the design's intent is Balance §5.
  • Exact vectors, not vibes. Pinning five levels (92/188/291/595/1811) means a subtly-wrong formula can't sneak through — a "greater than" check alone would let it.
  • One behavior per check, with got-vs-expected output, so a failure points at exactly what broke.
